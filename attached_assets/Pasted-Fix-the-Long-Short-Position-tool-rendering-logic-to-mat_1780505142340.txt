Fix the Long/Short Position tool rendering logic to match TradingView behavior exactly.

Current issue:
When Stop Loss is hit, the green TP/profit fill still remains visible. The position visualization logic is incorrect.

Implement TradingView-style layered rendering architecture.

Requirements:

1. Separate STATIC zones from LIVE progress visualization.

STATIC LAYERS:
- TP green rectangle (entry -> target)
- SL red rectangle (entry -> stop)
- These are always rendered as fixed background zones.

LIVE DYNAMIC LAYER:
- Separate overlay showing active running movement from entry price to current running price.
- This must NOT reuse the same TP rectangle.
- Render as independent dynamic layer.

2. Entry anchoring behavior:
- Position must start from the LEFT EDGE of the candle where the tool is placed.
- Not from candle center.
- All fills, entry lines, TP/SL regions must align with candle left edge exactly like TradingView.

3. Running visualization logic:
WHILE tradeStatus === "RUNNING":

- Measure from entry candle -> current running candle.
- Draw thick dynamic progress visualization.
- Use thick color fill and movement representation.

LONG POSITION:
- Current price above entry:
  -> dynamic GREEN movement
- Current price below entry:
  -> dynamic RED movement

SHORT POSITION:
- Reverse logic.

4. TP / SL completion behavior:

IF TP is hit:
- Freeze final profit state.
- Stop live updates.
- Remove active animated progress layer.
- Keep completed TP visualization.

IF SL is hit:
- Immediately remove all active green/live profit overlays.
- Freeze final red loss state.
- Stop all running progress rendering.
- Do NOT keep green active movement visible after SL hit.

5. State management:

Use explicit trade states:
- RUNNING
- TP_HIT
- SL_HIT

Example:

if (tradeStatus === "RUNNING") {
   renderLiveProgress();
}

if (price >= TP) {
   tradeStatus = "TP_HIT";
   clearLiveProgress();
   renderFinalTPState();
}

if (price <= SL) {
   tradeStatus = "SL_HIT";
   clearLiveProgress();
   clearGreenOverlays();
   renderFinalSLState();
}

6. Live movement style:
- Use thick TradingView-style movement visualization.
- Dynamic running area should update tick-by-tick.
- Visualization should stop exactly at the candle where TP or SL gets hit.
- Do not extend progress after trade completion.

7. Rendering architecture:
Create independent layers:

- staticTpLayer
- staticSlLayer
- liveProgressLayer
- labelsLayer
- handlesLayer

Never reuse static TP/SL rectangles as live movement overlays.

8. Final expected behavior:
- Exact TradingView-style long/short position tool behavior.
- Correct TP/SL completion handling.
- No stale green overlays after stop loss hit.
- Smooth real-time updates.
- MT5-style fast interaction and rendering performance.
